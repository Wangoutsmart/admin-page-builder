# 后台页面构建器

## 概述

使用这个 skill 来生成一种默认架构，用于标准后台页面。
要尽早约束请求 hooks、状态归属和弹窗控制方式，避免生成的代码在不同页面之间逐渐失控、风格漂移。

## 判断请求是否适用

只有当以下大部分条件都成立时，才将该请求视为一个标准后台页面：
	•	主体区域是一个带筛选和分页的列表或表格。
	•	常见操作是新建、编辑、详情、删除、启用、禁用，或其他轻量级的行级变更操作。
	•	页面可以由一个父级容器加若干弹窗或抽屉子组件来统一编排。
	•	页面不需要树形数据、多步骤流程状态、嵌套对话框，或跨区域复杂联动编排。

如果请求不符合这些条件，就不要继续使用这套默认配方。
应明确说明冲突点、列出缺失的项目规范，并避免临时再发明第二套模式。

## 输出约定

当请求适用时：
	1.	生成一套推荐的页面结构。
	2.	使用本 skill 中默认的请求和状态管理规则。
	3.	只指出那些依赖宿主项目约定的假设前提。

当请求不适用时：
	1.	解释为什么它超出了 v1 的适用范围。
	2.	询问缺失的业务规则或项目规范。
	3.	不要混入其他 hook 体系或状态管理模式。

## 请求策略

使用 useAntdTable 来处理列表请求，它负责整合筛选、分页和刷新行为。
使用 useRequest 来处理详情请求、新建/编辑提交、删除、启用/禁用、导出，或其他非表格类异步操作。
当页面同时包含表格和变更操作时，应采用一个 useAntdTable 搭配多个针对具体操作的小型 useRequest。

不要引入 react-query、useQuery、useMutation、全局异步 store，或第二套请求抽象，除非宿主项目本身已经强制要求使用。

## 状态归属

将筛选值、已选中行、当前记录、弹窗显隐状态，以及表格刷新动作统一放在父级页面中管理。
子组件通过 props 接收数据，并通过回调上报意图，例如 onSuccess、onCancel、onSelect 或 onClose。
不要让子组件直接打开其兄弟组件、直接刷新表格，或通过隐藏导入、单例、事件总线等方式偷偷修改页面状态。
优先采用单向数据流：

页面状态 -> 子组件 props -> 子组件回调 -> 父组件刷新

## 弹窗与抽屉规则

新建或编辑弹窗必须由父页面打开。
详情抽屉也必须由父页面打开。
向弹窗类子组件传入 open、mode、record 或 initialValues、以及 onSuccess 或 onCancel。
只有在操作成功后，或者用户主动取消时，才关闭弹窗。
新建、编辑、删除、启用、禁用等操作成功后，由父组件触发表格刷新。
嵌套弹窗、弹窗链路、弹窗套弹窗这类交互流，属于本 v1 skill 的范围之外。

## API 参数规则

请求参数应基于页面上可见的表单字段和查询字段来构建。
对于 operatorId、currentUserId、登录用户信息等操作者字段，只有在项目约定明确规定由后端、网关或请求拦截器自动补齐时，才可以在页面层忽略。
如果这个约定不明确，就应将其标记为例外情况，而不是自己猜。
参数整形逻辑要尽量靠近请求调用处，不要分散到表格列定义、表单组件和子组件里。

## 交付规则

让成功路径尽量简单：

## 提交 -> 按项目规范提示成功 -> 关闭弹窗 -> 刷新表格

请求失败的处理应局限在失败的那个操作自身。
除非宿主项目已经提供了稳定的基础表格、弹窗表单或请求封装，否则优先采用页面内局部组合，而不是抽象出共享层。
