# 表单规范

## 说明

使用这份规范统一处理中后台项目中的表单布局、校验、控件表现和复用方式。
默认让表单表现稳定一致，只根据真实业务约束生成校验规则，不要机械补规则或在同类表单里写出多套不同风格。

## 布局规范

- 在 `Form` 组件上统一声明 `labelCol`，不要把 label 宽度分散写在单个 `Form.Item` 上。
- `labelCol.span` 要参考当前表单中最长的可见 label 长度来确定，保证整体对齐效果，不要随手写一套固定值。
- `labelCol` 必须兼顾大屏和小屏的弹性展示，不要只写一套桌面端固定 span。
- 同一张表单中的布局规则保持一致，不要让部分字段左侧对齐、部分字段另起一套 label 宽度。
- 只有在明显独立的分区表单中，才允许按分区重新定义布局。

例如：

```tsx
<Form
  labelCol={{
    xs: { span: 24 },
    sm: { span: 8 },
    md: { span: 6 },
    lg: { span: 5 },
  }}
>
```

## 控件宽度规范

- 表单中的输入类控件默认保持统一宽度，不要因为组件默认行为让 `InputNumber`、`Select`、`DatePicker` 明显窄于 `Input`。
- `Input`、`InputNumber`、`Select`、`DatePicker`、`Cascader`、`TreeSelect` 等表单控件，默认按表单标准宽度渲染，优先使用 `style={{ width: '100%' }}` 或项目内统一的宽度封装方式。
- 只有在业务明确要求固定宽度、内容自适应宽度，或同一行内需要多个字段并排时，才允许使用非全宽样式。
- 同一行内多个字段并排时，也要保证整体视觉对齐，不要单独让某个控件看起来“短一截”。

## 校验规则规范

- Form.Item.rules 必须根据真实约束生成，不要机械补默认校验。
- 当用户需求、产品文案或设计稿明确说明字段必填，或者字段有红色 \* 标识时，必须补充必填校验。
- 搜索表单默认不要补必填规则，只有业务明确要求时才添加。
- 当字段存在“只能输入数字”“最多保留两位小数”“长度限制”“格式限制”这类明确约束时，才补充对应的 rules。
- 当 `InputNumber`、原生组件属性或现有控件能力不能完整覆盖限制时，再补 `pattern` 校验。
- `pattern`、`min`、`max`、`len`、`validator` 等规则必须服务于真实业务约束，不要为了“看起来规范”而默认添加。
- 如果原生组件能力已经完整限制输入，就不要再补一层等价规则。
- 校验文案要表达真实业务含义，不要堆砌空泛提示。

## 输入处理规范

- 普通文本输入默认在失焦或提交前自动过滤前后空格。
- 表单校验和提交都基于去除前后空格后的值处理，不要直接使用原始输入值。
- 只有在业务明确要求保留前后空格时，才跳过这条规则，例如密码、密钥、富文本、代码片段或其他格式敏感内容。
- 不要为了处理空格问题，在提交函数里到处临时`trim`；优先在表单层统一处理。

## 新增编辑查看复用规范

- 当新增、编辑、查看页面的表单结构、字段布局和交互表现基本一致时，优先复用同一个表单组件，不要为 `create`、`edit`、`view` 分别拆三套重复实现。
- 复用组件通过 `mode` 区分当前场景，例如 `create`、`edit`、`view`，并根据 `mode` 控制标题、默认值、禁用态、提交按钮和页面文案。
- 如果只是字段回填、禁用状态、按钮文案或提交逻辑不同，应该在同一个组件内收敛处理，不要因此拆出多个近似页面。
- 只有在字段集合、页面布局或交互流程明显分叉时，才允许拆成独立表单组件。
- 如果一个共享表单已经出现大量 `if (mode === ...)` 分支，并且结构和逻辑都明显分叉，应及时拆分，避免为了复用而复用。

## 搜索表单规范

- 搜索表单也应遵守统一的布局和控件宽度规则。
- 搜索表单默认以“易录入、易重置”为目标，不要照搬新增/编辑表单的强校验策略。
- 搜索表单字段只保留真正用于查询的项，不要把详情字段或提交字段直接搬过来。
- 搜索表单中的日期区间、状态选择、下拉筛选等字段，仍然应保持和主表单一致的控件宽度与对齐方式。

## 组件表现规范

- 表单场景中的 `Select`、`InputNumber`、`DatePicker` 等控件，既要遵守各自组件规范，也要首先服从当前表单的布局和宽度规范。
- 如果某个控件存在默认表现与表单整体风格不一致，应优先在表单层统一修正，而不是让每个字段各自处理。
- 查看态如果只是“同结构不可编辑”，优先通过`mode + disabled / readOnly`收敛，不要额外写一套纯展示表单。
- 查看态如果不是简单禁用，而是完全不同的信息展示结构，才允许拆成独立详情组件。
