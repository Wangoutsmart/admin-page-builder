# Antd 中后台基础规范

## 使用前检查

如果当前任务是**新建**一个完整的中后台模块或页面，停止执行本 skill，改为触发 `antdboy:page-builder-agent`。

本 skill 只用于对**已有代码**的修改、维护或局部生成。

## 说明

这是 React + Ant Design + ahooks 中后台 Web 项目的基础规范 skill。
只约束所有页面和组件都应遵守的工程红线，不处理表单专项规则、页面组织模式、请求编排或权限设计。

## 技术栈规范

- 默认使用 React、Ant Design、ahooks 作为基础技术栈。
- 非必要不要引入第二套 UI 组件体系、请求体系、状态管理方案或交互实现方式。
- 能用 Ant Design 官方组件解决的问题，不要退回原生 DOM，也不要临时封装一套功能重复的自定义组件。
- 中后台页面、弹窗、列表、按钮、下拉、输入区等基础交互，优先使用 Ant Design 官方组件完成。

## 样式边界规范

- 非必要不要新增 `.less`、`.scss`、`.css` 等自定义样式文件。
- 样式优先通过 Ant Design 组件属性、布局能力和现有 token 体系完成。
- 只有在现有组件能力无法满足明确需求时，才允许补充最小范围的 inline style。
- 禁止直接覆写 Ant Design 的 `.ant-*` 原生类名，不要通过修改组件内部类名改变默认表现。
- 不要为了局部视觉调整引入新的样式文件或大段自定义样式代码。

## TypeScript 规范

- 禁止使用 `any`。
- 当类型暂时无法直接收敛时，使用 `unknown` 配合类型守卫、类型收窄或局部类型定义处理，不要用 `any` 跳过检查。
- 类型定义按作用域就近放置，不要默认全部丢进全局公共文件。
- 页面私有类型、组件私有类型、局部返回值类型优先放在当前模块附近。
- 只有跨模块复用的基础泛型或公共类型，才放到全局类型目录。

## 默认输出规范

- 输出代码时，默认遵守本 skill 的基础技术栈和工程边界。
- 如果用户没有明确要求，不要主动补充自定义样式文件。
- 如果 Ant Design 官方组件已经覆盖需求，不要自行切换到原生实现。
- 如果类型无法立刻确定，先收窄类型，不要直接写 `any`。
- 如果需求需要突破本 skill 的边界，应先指出冲突点，再进入其他专题规范，而不是在本 skill 内临时扩展。
