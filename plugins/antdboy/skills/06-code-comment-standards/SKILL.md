# 代码注释规范

## 说明

使用这份规范统一处理中后台前端代码中的注释风格和注释边界。
默认让注释只出现在真正有信息量的地方，不要为了“看起来规范”而机械补注释。

## 注释语言规范

- 只写中文注释，不写英文注释。
- `TODO`、`FIXME`、`HACK`、`NOTE` 这类标记，也使用中文语义表达，例如 `待处理`、`待修复`、`临时方案`、`注意`。
- 代码标识、组件名、hook 名、接口字段名保留原始英文，不要强行翻译到注释里。
- 注释文案保持简洁直接，不写口语化说明，不写无意义修饰语。

## 注释出现时机

- 简单直白的代码不要写注释，例如变量赋值、普通条件渲染、标准 CRUD 提交、直接的 props 透传。
- 只有在下面这些情况出现时，才允许写注释，并且注释必须简短：

1. 业务规则不直观  
   例如状态映射、权限分支、后端字段兼容、特殊值约定。
2. 存在明显的边界条件或易错点  
   例如空值兜底、索引处理、时间区间、精度问题、重复提交保护。
3. 代码实现不是最自然写法  
   例如为了兼容旧接口、绕过三方组件问题、按性能要求拆成两步处理。
4. 有跨组件、跨层级、跨请求的隐式约束  
   例如“这里不能直接刷新列表，需要等弹窗成功回调后由父组件统一刷新”。
5. 异步流程较难一眼看懂  
   例如串行请求、竞态取消、重试、防抖、提交锁。

## 注释表达规范

- 注释优先解释“为什么这样写”或“不这样写会怎样”，不要复述代码表面行为。
- 不要写“设置 loading 为 true”“把数据赋值给变量”这类从代码字面就能看出的注释。
- 注释长度控制在 1 到 2 行，只有在解释必要业务前提时才允许稍长。
- 如果一段代码需要长篇注释才能解释清楚，优先先拆函数、重命名变量或提炼小组件，再决定是否补注释。
- 注释应尽量贴近其解释的代码，不要把说明写得离实际逻辑过远。

## swagger接口文档类型定义注释规范

- 根据 Swagger、OpenAPI 等接口文档生成 TypeScript 类型定义时，如果文档中的字段、模型或接口说明包含中文注释或中文描述，应在生成的类型定义中保留对应的中文注释。
- 中文注释应尽量直接对应字段或类型本身的业务含义，不要省略已有中文说明，也不要自行改写成英文注释。
- 类型名称、字段名称保持原始接口命名规则，中文语义通过注释补充，不要把字段名强行翻译成中文。
- 字段级中文说明写在对应属性上方或同行，类型级中文说明写在类型定义上方。
- 如果接口文档中没有有效中文说明，不要为了补注释而自行编造业务含义。
- 如果同一个字段同时存在名称、类型、中文说明，优先保留原始字段名和类型定义，再用中文注释补充业务解释。

## 不推荐的注释

- 不要给每个变量、每个 `if`、每个 JSX 结构都补注释。
- 不要把注释写成需求文档、接口文档或操作说明。
- 不要使用无信息量注释，例如“这里做校验”“这里处理数据”“这里发请求”。
- 不要在代码已经能清晰表达语义时，再写一层重复注释。
